# P3 简历段（STAR）

## 一句话版（简历用）

> **多模态 IDC 应标书智能审核系统**：VLM 读图 + docx 解析 + 多模态 RAG，按 checklist 自动审核应标书响应完整性与技术达标性；通过四层"假缺失"排查系统性修复检索管道，审核结果达 12/12 出处可追溯、0 幻觉。

## STAR 版（面试展开）

**S（场景）**：中国移动 IDC 实习期间，人工审核应标技术标书是重复且吃业务经验的活——逐条核对招标要求（机房等级/SLA/供电冗余/偏离表响应）。标书几百页、含几百张拓扑图与机柜布局图，人工审慢且易漏。

**T（任务）**：做一个能自动审核应标书的系统，不止读文本、还要读图（应标书信息以图为核心），按可配置 checklist 输出分维度审核报告，并对结果做诚实评估。

**A（行动）**：
- 设计五环 pipeline：docx 解析（python-docx + 解压内嵌图）→ 多模态入库（图经 VLM/qwen-vl-max 转描述再 embedding、metadata 拴原图）→ 定向检索 → 审核 Agent（结构化输出 + 防幻觉 prompt）→ Jinja2 报告。
- 首版报告异常（12 项废标红线 9 项缺失），逐层排查证据挖出**四层"假缺失"**并修复：①库没入全量（分批入库）②k 太小碎片挤占（k=4→8）③query 词汇鸿沟（行业同义词扩展）④图污染文本窗口（按维度定向检索）。
- 建三轴评估（召回诊断 / 忠实度抗幻觉 / 专家判断框架），诚实标注边界。
- Gradio 前端 + Docker 一键部署；真实标书全程本地处理，零敏感数据入公开仓。

**R（结果）**：
- 端到端 pipeline 打通，缺失误判从 9 项降到 2 项（其余为标书真实情况）。
- 忠实度 **12/12 出处可追溯、0 幻觉**（防幻觉铁律通过）。
- 沉淀"检索为空 ≠ 内容缺失"的排雷方法论，覆盖 RAG 数据管道从入库到多模态干扰的四类典型陷阱。

## 面试预备追问

- **为什么应标书比招标书图多？** 乙方要秀方案（拓扑/机柜布局），图是核心；招标书甲方提要求，多文字。多模态项目必须选图多的场景。
- **图怎么进向量库？** embedding 只吃文字 → 图先过 VLM 转结构化描述 → embedding，metadata 存 image_path 拴原图，检索命中描述后顺回原图。
- **加同义词是不是过拟合？** 用行业通用同义词（可用性↔业务连续性）是正当检索工程；反向抠标书原文塞 query 才是作弊。守住了这条线。
- **0 幻觉可信吗？** 忠实度定义为"出处在知识库真实存在"而非某次检索窗口；2 个红旗查实是校验脚本自身 bug，不是真幻觉。
- **为什么不报准确率？** 审核任务无 gold answer，真实漏检率需专家读整份标书；样本 1 份，只作方法论验证，不宣称统计意义。
- **最大的坑？** 四层假缺失——尤其"图污染文本检索窗口"是纯多模态特有，P1/P2 遇不到。
