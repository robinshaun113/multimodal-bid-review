# P3 简历段（STAR）

## 一句话版（简历用）

> **多模态 IDC 招投标合规审核系统**：从招标 DOCX 动态抽取强制要求，在应标文本、表格和图片中检索证据，以 LangGraph 并行生成 Requirement×Evidence 合规矩阵；通过四层“假缺失”排查修复检索管道，并用 evidence_id 将引用约束到单次审核窗口。

## STAR 版（面试展开）

**S（场景）**：中国移动 IDC 实习期间，人工审核应标技术标书是重复且吃业务经验的活——逐条核对招标要求（机房等级/SLA/供电冗余/偏离表响应）。标书几百页、含几百张拓扑图与机柜布局图，人工审慢且易漏。

**T（任务）**：构建招标—应标双文档审核系统：从招标文件抽取强制项和阈值，在应标文本/表格/图片中匹配证据，输出可追溯合规矩阵，并将废标风险交给人工确认。

**A（行动）**：
- 设计双文档 pipeline：保序解析段落/表格并记录图片 OOXML 锚点 → Qwen3.7 结构化抽取 Requirement → 应标证据独立索引 → LangGraph `Send` 并行审核/重试 → 废标项 HITL → Jinja2 报告。
- 首版报告异常（12 项废标红线 9 项缺失），逐层排查证据挖出**四层"假缺失"**并修复：①库没入全量（分批入库）②k 太小碎片挤占（k=4→8）③query 词汇鸿沟（行业同义词扩展）④图污染文本窗口（按维度定向检索）。
- 将文本/图片统一为稳定 evidence_id，LLM 只能引用当次检索窗口；对“≥99.99%”“≤30 分钟”等唯一数值阈值增加确定性规则复核，歧义场景不强判，区分引用可追溯、检索召回和最终判断准确率。
- 增加 qwen3-vl-embedding 原生以文搜图实验轨道，与图片描述基线独立评测；Gradio + Docker 完成双文件演示。

**R（结果）**：
- 端到端 pipeline 从单份行业基准扫描升级为招标—应标 Requirement×Evidence 对照审核，并通过 11 个无模型测试覆盖文档顺序、需求溯源、引用窗口、数值规则、并行路由与原生多模态响应契约。
- 旧样本上系统输出的“缺失”项由 9 项降至 2 项；该数字仅说明管道修复前后变化，不等同审核准确率。
- 沉淀"检索为空 ≠ 内容缺失"的排雷方法论，覆盖 RAG 数据管道从入库到多模态干扰的四类典型陷阱。

## 面试预备追问

- **为什么应标书比招标书图多？** 乙方要秀方案（拓扑/机柜布局），图是核心；招标书甲方提要求，多文字。多模态项目必须选图多的场景。
- **图怎么进向量库？** 保留两条轨道：VLM 描述→文本向量作为稳定基线；qwen3-vl-embedding 直接将图与文本放入同一语义空间，先用图片 Recall@K 评测再决定是否默认融合。
- **加同义词是不是过拟合？** 用行业通用同义词（可用性↔业务连续性）是正当检索工程；反向抠标书原文塞 query 才是作弊。守住了这条线。
- **怎么证明引用可信？** 每项审核保存 retrieved_evidence_ids，输出 citation 必须属于该集合且 quote 命中对应 evidence；该检查只能证明引用受证据约束，不宣称“零幻觉”。
- **为什么不报准确率？** 审核任务无 gold answer，真实漏检率需专家读整份标书；样本 1 份，只作方法论验证，不宣称统计意义。
- **最大的坑？** 四层假缺失——尤其"图污染文本检索窗口"是纯多模态特有，P1/P2 遇不到。
