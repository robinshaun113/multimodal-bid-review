# 多模态 RAG 在 IDC 应标书审核中的落地实验

> 一份技术博客，讲的不是"我搭了个多模态系统"，而是"一个假报告是怎么被一层层挖穿的"。

## 从一个真实痛点说起

在中国移动 IDC 实习期间，我做过一件重复且吃业务经验的活：拿招标要求逐条核对应标技术标书——机房等级够不够国标 A 级、SLA 有没有承诺到 99.9%、供电是不是 N+1 冗余、技术规格偏离表有没有逐条正面响应。一份标书几百页、几百张图，人工审又慢又漏。

我想用 AI 接住它。但选场景时踩了个关键判断：**招标书天生图少**（甲方提要求，多文字），**应标书天生图多**（乙方秀方案，含大量拓扑图、机柜布局图）。多模态项目图是灵魂，所以必须做**应标书审核**——纯文本 RAG 接不住那几百张图。

## 五环 pipeline

```
docx → ①解析 → ②多模态入库 → ③检索 → ④审核Agent → ⑤报告
```

- **①解析**：docx 本质是 zip，文本用 python-docx 抽（段落+表格），图直接解压 `word/media/`。比扫描件 PDF 结构化得多。
- **②多模态入库**：embedding 只吃文字，图不能直接入库——图先过 VLM(qwen-vl-max) 读成文字描述，再 embedding，metadata 存 `image_path` 拴住原图。
- **③检索 + ④审核**：每条 checklist 拿 query 检索证据，连同 expect 标准喂 LLM，`with_structured_output` 判达标/风险/缺失。防幻觉铁律：只依据检索原文，检索不到就判缺失，绝不编造。
- **⑤报告**：Jinja2 渲染，结论摘要钉顶、废标红线优先。

## 真正的主线：四层"假缺失"

第一版报告：**12 项废标红线里 9 项缺失**。一份中标级标书不可能全崩——异常，查。

### 第一层：库根本没入全量

查向量库：只有 **25 条**（早期链路验证用的 15 文本 + 10 图小样本）。而真实标书解析出 **4011 文本块 + 440 张图**。审核 Agent 在残缺库上跑，满屏"检索内容中未提及"——这不是标书缺，是**内容没进库**。

修：全量文本入库。踩坑：DashScope `text-embedding-v2` 兼容模式单次 batch 上限 25 条，4011 条不能一次塞，用 `chunk_size` + 分批 add 绕过。缺失 **9 → 5**。

### 第二层：k 太小，碎片挤占窗口

旧版全量库里 3236 条文本，很多是碎片——当时 docx 表格按行切，"合计 | 22.9"这种单行无语义。审核用 k=4 检索，query 是关键词串，**高度匹配的标题/表头挤占前排**，实质内容被挤到 k=5 之后；新版已改为完整表格块保序解析。

反直觉的点：**库数据越多，某些内容反而越难召回**。修：k=4 → 8。缺失 **5 → 2**。

### 第三层：query-document 词汇鸿沟

SLA、制冷两项，k 提到 15 还是缺失。但手换措辞就能捞到——根因是 **query 措辞**。checklist 写「SLA 服务可用性」，标书写「业务连续性(可靠性)不低于 99.99%」——**这句压根没"SLA"这个词**，语义 embedding 跨不过这个鸿沟；更糟的是"SLA"反把检索带偏到"SLA 管理流程"。

修：query 用 **IDC 行业同义词**扩展（可用性↔业务连续性/可靠性）。

**这里有条诚信线**：用行业通用同义词是正当的检索工程；反向把标书原文抠出来塞进 query 才是过拟合作弊。k 没动仍 =8，只改措辞就翻正。

### 第四层：图污染文本检索窗口（多模态特有）

修完前三层，把 440 张图抽样入库救活图纸维度后，一个文本项——罚则条款——**从"达标"退步成"缺失"**。查检索窗口：k=8 里挤进了 2 张图描述，把原本证明"约定违约责任"的文本证据挤出去了。

这是纯多模态系统才有的坑：**图入库救活了图纸维度，却反过来污染了文本项的检索窗口**。修：按维度定向检索——审罚则/报价不该看拓扑图，只有"图纸完整性"维度才检索图。

### 四层串起来的认知

| 层 | 根因 | 修复 |
|---|---|---|
| ① | 库没入全量 | 分批全量入库 |
| ② | k 太小，碎片挤占 | k=4→8 |
| ③ | query 词汇鸿沟 | 行业同义词扩展 |
| ④ | 图污染文本窗口 | 按维度定向检索 |

**检索为空 ≠ 内容缺失。** 做审核/评估前，必须验证数据管道每一环（入库量→召回质量→query 措辞→模态干扰），否则报告再漂亮也是建在空库上。

## 评估：不编准确率

审核任务没有 gold answer，真实漏检率要专家读整份标书。我没编个"92%"贴上去，而是分三轴诚实标注：

- **召回诊断**（全自动）：k=8 证据 vs 深捞 k=25，12 项中 10 项 k=8 外仍有关键证据，标出高危复核点。
- **引用约束**（全自动）：新版为每个证据块建立稳定 `evidence_id`，保存每项审核实际看到的 `retrieved_evidence_ids`；引用必须属于该窗口且短句命中对应证据。旧版“12/12 短引用可在全库找到”只作迁移基线，不再表述为“0 幻觉”。
- **对错判断**：搭框架（专家填 CSV → 算一致率/漏检率/误检率），留给领域专家。

诚实边界：引用可追溯不等于判断正确；样本 1 份，方法论验证级，不宣称统计意义。

## 已知局限（不藏着）

- 拓扑图多为 emf 矢量格式，qwen-vl-max 不支持，需先转 png——当前拓扑维度靠文本判断，没完全落地。
- DOCX 解析已保留图片 OOXML 锚点和所属章节，但流式排版下仍没有可靠页码；如需页码级审计应先渲染 PDF，再接布局解析/OCR。
- 表格已按完整表格块保序解析；合并单元格、跨页表头和复杂嵌套仍需专门布局测试。

## 小结

这个项目我最想留下的不是几百行代码，是**"让证据指路、层层查根因"的工程习惯**：一个反常的报告，别急着调参数，先问"数据管道哪一环出了问题"。四层假缺失，每一层都是这么挖出来的。
