# 多模态 RAG 在 IDC 应标书审核中的落地实验

> 这篇记录重点介绍一次检索管道排查：为什么系统会把已有内容判断为缺失，以及如何逐层验证。

## 从一个真实痛点说起

在梳理 IDC 解决方案和投标材料时，我注意到标书审核需要逐条核对机房等级、SLA、
供电冗余和技术规格偏离表。一份标书可能包含数百页文本和大量拓扑图，单纯依赖关键词
搜索很难覆盖图文混合证据。

因此我选择“招标要求—应标证据”对照作为项目场景：招标文件负责提供要求，应标文件
负责提供文本、表格和图片证据。纯文本 RAG 无法直接处理拓扑图和机柜布局图，因此需要
额外的图片理解与跨模态检索链路。

## 五环 pipeline

```
①招标需求抽取 → ②应标多模态证据索引 → ③检索 → ④审核 Agent → ⑤报告
```

- **①招标需求抽取**：保序解析 DOCX，通过结构化输出提取 Requirement，并校验要求原文确实存在于对应招标块。
- **②应标多模态入库**：文本和表格直接建立索引；图片主链路先由 qwen-vl-max 生成描述，再进入文本向量库，并通过 metadata 关联原图。原生多模态 embedding 作为独立实验轨道。
- **③检索 + ④审核**：每条 Requirement 检索应标证据，使用 `with_structured_output` 输出达标/风险/缺失；引用必须属于本次检索窗口。
- **⑤报告**：Jinja2 渲染合规矩阵，摘要和废标红线优先展示。

## 关键排查：四类“假缺失”

第一版报告显示 **12 项废标红线里有 9 项缺失**。这个结果与文档内容明显不符，因此我从入库、召回、查询和模态四层分别检查。

### 第一层：向量库未包含全量内容

查向量库：只有 **25 条**（早期链路验证用的 15 文本 + 10 图小样本）。而真实标书解析出 **4011 文本块 + 440 张图**。审核 Agent 在残缺库上跑，满屏"检索内容中未提及"——这不是标书缺，是**内容没进库**。

修正方式是全量文本入库。DashScope `text-embedding-v2` 兼容模式存在 batch 上限，
因此使用 `chunk_size` 和分批 `add_documents`。历史样本中的缺失输出从 **9 降到 5**。

### 第二层：k 太小，碎片挤占窗口

旧版全量库里 3236 条文本，很多是碎片——当时 docx 表格按行切，"合计 | 22.9"这种单行无语义。审核用 k=4 检索，query 是关键词串，**高度匹配的标题/表头挤占前排**，实质内容被挤到 k=5 之后；新版已改为完整表格块保序解析。

反直觉的点：**库数据越多，某些内容反而越难召回**。修：k=4 → 8。缺失 **5 → 2**。

### 第三层：query-document 词汇鸿沟

SLA、制冷两项，k 提到 15 还是缺失。但手换措辞就能捞到——根因是 **query 措辞**。checklist 写「SLA 服务可用性」，标书写「业务连续性(可靠性)不低于 99.99%」——**这句压根没"SLA"这个词**，语义 embedding 跨不过这个鸿沟；更糟的是"SLA"反把检索带偏到"SLA 管理流程"。

修：query 用 **IDC 行业同义词**扩展（可用性↔业务连续性/可靠性）。

同义词只使用可复用的 IDC 行业表达，没有把目标标书原句直接写入 query；`k` 保持为 8，
以区分查询改写收益和检索窗口变化。

### 第四层：图污染文本检索窗口（多模态特有）

修完前三层，把 440 张图抽样入库救活图纸维度后，一个文本项——罚则条款——**从"达标"退步成"缺失"**。查检索窗口：k=8 里挤进了 2 张图描述，把原本证明"约定违约责任"的文本证据挤出去了。

这是多模态索引带来的新问题：图片描述提升了图纸类召回，却挤占了文本要求的检索窗口。
修正方式是按维度设置 metadata filter，只有图纸类要求同时检索图片证据。

### 四层串起来的认知

| 层 | 根因 | 修复 |
|---|---|---|
| ① | 库没入全量 | 分批全量入库 |
| ② | k 太小，碎片挤占 | k=4→8 |
| ③ | query 词汇鸿沟 | 行业同义词扩展 |
| ④ | 图污染文本窗口 | 按维度定向检索 |

**检索为空 ≠ 内容缺失。** 做审核/评估前，必须验证数据管道每一环（入库量→召回质量→query 措辞→模态干扰），否则报告再漂亮也是建在空库上。

## 评测口径

审核任务目前没有专家标注的 gold answer，因此我将可验证部分拆成三层：

- **召回诊断**（全自动）：k=8 证据 vs 深捞 k=25，12 项中 10 项 k=8 外仍有关键证据，标出高危复核点。
- **引用约束**（全自动）：新版为每个证据块建立稳定 `evidence_id`，保存每项审核实际看到的 `retrieved_evidence_ids`；引用必须属于该窗口且短句命中对应证据。旧版“12/12 短引用可在全库找到”只作迁移基线，不再表述为“0 幻觉”。
- **对错判断**：保留人工 CSV 标注框架，用于计算一致率、漏检率和误报率。

引用可追溯不等于判断正确；当前样本量只适合验证方法，不具有统计意义。

## 已知局限

- 拓扑图多为 emf 矢量格式，qwen-vl-max 不支持，需先转 png——当前拓扑维度靠文本判断，没完全落地。
- DOCX 解析已保留图片 OOXML 锚点和所属章节，但流式排版下仍没有可靠页码；如需页码级审计应先渲染 PDF，再接布局解析/OCR。
- 表格已按完整表格块保序解析；合并单元格、跨页表头和复杂嵌套仍需专门布局测试。

## 小结

这次排查让我确认，合规审核中的“缺失”不能只看模型结论。出现异常报告时，应先验证
文档是否完整入库、关键证据能否召回、query 是否覆盖行业表达，以及不同模态是否相互
挤占检索窗口，再讨论模型判断质量。
